# Conduit — Refactor Context

> Architecture snapshot generated by crawling the codebase on 2026-04-24.
> Intended as input to a planned refactor. Not a design doc, not a roadmap —
> a map of what's here, what couples to what, and where the smell is.
>
> Verify any file/line reference before acting on it; this is a snapshot.

## 1. Scope

Conduit is a Dart HTTP server framework, forked from Stablekernel's Aqueduct.
Current workspace version is **6.0.0**, Dart SDK `>=3.11.0 <4.0.0`, managed
with Melos. Monorepo at `packages/*`. The user-facing entry point is the
`conduit` CLI (`packages/cli`) and the `conduit_core` library
(`packages/core`).

Two axes dominate the architecture:

1. **Aqueduct lineage**: reflection-heavy (`dart:mirrors`) design from a
   pre-AOT Dart world.
2. **Post-fork AOT story**: a build-time codegen pipeline
   (`packages/runtime`) that deflects mirrors out so a production binary can
   run without them.

Almost every refactor decision in this codebase is a point on the tension
between those two.

Observations:
- `core` sits on top of almost everything.
- `cli` duplicates most of core's deps because it spawns isolates that import
  user code through core's types (ORM, auth, OpenAPI).
- `common` is a one-file bridge between `core`, `cli`, and `open_api`. It is
  load-bearing and has zero test coverage.
- `codable` is the only leaf package with no Conduit deps — safe to treat as
  a stable primitive.

## 3. Per-package architecture

### 3.1 `packages/core` — framework

Top-level export: [lib/conduit_core.dart](packages/core/lib/conduit_core.dart).
Secondary entry: [lib/managed_auth.dart](packages/core/lib/managed_auth.dart) (OAuth tables as `ManagedObject`s).

Subsystems under `lib/src/`:

- **application/** — lifecycle, isolates, HTTP server bootstrap.
  `Application<T>`, `ApplicationChannel`, `ApplicationServer`,
  `IsolateSupervisor`, `ApplicationMessageHub`. Heavy responsibility
  concentration in `Application`.
- **http/** — routing, controllers, middleware, serialization.
  `Controller`, `Router`, `RouteNode`, `RouteSpecification`,
  `ResourceController` (declarative REST via `@Operation` / `@Bind`),
  `ManagedObjectController`, `Request`, `Response`, `Serializable`,
  `CORSPolicy`, `CachePolicy`, `HttpCodecRepository`. `@Operation` caps
  path variables at 4 by virtue of named-parameter count.
- **auth/** — OAuth 2.0 authorization server (`AuthServer`), token / code /
  client DTOs, `Authorizer` middleware, `AuthController` /
  `AuthCodeController` / `AuthRedirectController`, parser interface for
  Basic/Bearer. Storage is delegated to `AuthServerDelegate`.
- **db/managed/** — ORM model registry. `ManagedObject<T>`,
  `ManagedEntity`, `ManagedDataModel` (two ctors: explicit types and
  `fromCurrentMirrorSystem()`), `ManagedContext`, `ManagedBacking`,
  `ManagedPropertyDescription` hierarchy, `@Column`, `@primaryKey`,
  `@Relate`, `@Transient`, `@Validate`.
- **db/query/** — fluent query DSL. `Query<T>`, `QueryMixin`,
  `Predicate`, `MatcherExpression`, `SortPredicate`, `PageDescriptor`,
  `Reduce`. Uses closure analysis (`query.where((o) => o.field)`) to
  recover column names — performance hazard, maintenance hazard.
- **db/schema/** — schema snapshot + diff + migration code generation.
  `Schema`, `SchemaTable`, `SchemaColumn`, `SchemaBuilder`, `Migration`.
  Driven by filename-convention discovery.
- **db/persistent_store/** — `PersistentStore` interface. Only live
  implementation is `conduit_postgresql`.
- **runtime/** — the bridge into `packages/runtime`. `ConduitCompiler`
  discovers `ApplicationChannel` / `Controller` / `Serializable` /
  `ManagedObject` subclasses and emits AOT-safe code. Mirror-backed
  dev-mode impls: `ChannelRuntimeImpl`, `ControllerRuntimeImpl`,
  `SerializableRuntimeImpl`, `ManagedEntityRuntimeImpl`. ORM compiler
  subtree (`runtime/orm/`): `DataModelCompiler`, `EntityBuilder`,
  `PropertyBuilder`, `ValidatorBuilder`, `EntityMirrors`.
- **utilities/** — `mirror_helpers.dart`, `sourcify.dart`. Small, under-documented.

Cross-layer couplings worth naming:
- `auth/` is a Controller subsystem, so it imports `http/`.
- `http/` imports `auth/` for Authorizer-aware routing.
- `managed_auth.dart` sits on top of `auth/` + `db/managed/`.
- `db/` and `http/` both import `runtime/` (for the compiler entry point).

### 3.2 `packages/cli` — `conduit` binary

Bootstrap: [bin/conduit.dart](packages/cli/bin/conduit.dart) → `Runner` at
[lib/src/runner.dart](packages/cli/lib/src/runner.dart). Subcommands:
`create`, `serve`, `build`, `document` (+ `document serve`), `db`
(generate / upgrade / validate / version / show-migrations / schema),
`auth`, `setup`.

Layout under `lib/src/`:

- `command.dart` — `CLICommand` base; argument discovery uses `dart:mirrors`
  against `@Flag` / `@Option` / `@MultiOption` annotations on every
  invocation.
- `commands/` — one file per subcommand.
- `mixins/` — `CLIProject` (pubspec + package_config discovery),
  `CLIDatabaseManagingCommand`, `CLIDatabaseConnectingCommand`,
  `CLIDocumentOptions`.
- `scripts/` — `Executable<T>` subclasses run inside `IsolateExecutor` for
  every code-gen task: `get_channel_type`, `get_schema`,
  `migration_builder`, `openapi_builder`, `run_upgrade`, `schema_builder`.
  These reach back into `conduit_core` types, which is why CLI pulls the
  full framework dep tree.
- `templates/` — scaffolds for `conduit create` (`default`, `db`,
  `db_and_auth`). Applied by naive string substitution (`wildfire` →
  project name).

`serve` spawns the user's app via `Isolate.spawnUri(dataUri, ...)` with a
generated startup script. `build` drives `BuildManager` from
`conduit_runtime`. Pub/psql are shelled out via `Process.run`.

### 3.3 `packages/runtime` — AOT compiler + mirror abstraction

Entry: [lib/runtime.dart](packages/runtime/lib/runtime.dart). Key types:

- `RuntimeContext` (abstract) with two implementations:
  - `MirrorContext` — dev mode, discovers `Compiler` subclasses via mirrors.
  - `GeneratedContext` — produced at build time; no mirrors at runtime.
- `Compiler` / `SourceCompiler` — plugin interface. `core/runtime`
  registers `ConduitCompiler`; `config` registers its own compiler.
- `BuildContext`, `BuildManager`, `Build` — orchestrate the pipeline:
  strip `main()` from user script → resolve AST → run each compiler's
  `getUrisToResolve` → emit per-runtime Dart files into
  `generated_runtime/` → call each compiler's `deflectPackage` to copy
  packages with mirrors imports rewritten → assemble final executable
  pubspec.
- `slow_coerce.dart` — hand-rolled, mirrors-free type coercion used by the
  generated context. Dispatches by **string prefix** on type name, which is
  brittle across Dart versions and doesn't cover user types.
- `mirror_coerce.dart` — mirror-based coercion, dev-mode fallback.

### 3.4 Supporting packages (short)

- **codable** — `Coding`, `KeyedArchive`, `Cast`. Clean, zero deps, fine.
- **common** — single file `src/openapi/documentable.dart` exporting
  `APIComponentDocumenter` / `APIOperationDocumenter`. **No tests.** Bridges
  `core` + `cli` → `open_api`.
- **config** — `Configuration`, mirror-backed parser + AOT compiler via
  `conduit_runtime`. Used by `core` and `cli`.
- **isolate_exec** — `Executor`, `Executable`, `SourceGenerator`
  (AST-driven closure dependency extraction). Clean.
- **open_api** — `v3/` (~18 files) + `v2/` (~13 files, Swagger). v2 is
  carried forward; unclear if anything still emits it — see §6.
- **password_hash** — `PBKDF2`, `Salt`. Single-purpose, used by auth.
- **postgresql** — the only live `PersistentStore` backend. Key files:
  `postgresql_persistent_store.dart`, `postgresql_query.dart`,
  `postgresql_query_reduce.dart`, `postgresql_schema_generator.dart`,
  `query_builder.dart`, `row_instantiator.dart`. Historical MySQL / SQLite
  mentions in docs but no in-tree driver.
- **test_harness** — `TestAgent`, `TestHarness`, `TestClient`, matchers,
  `AuthHarness`, `DBHarness`. Well-factored.
- **fs_test_agent** — `WorkingDirectoryAgent`, `DartProjectAgent`. Dev-only.

## 4. Cross-cutting concerns

### 4.1 Mirrors usage (the central refactor axis)

`dart:mirrors` is invoked in several independent places:

| Site | Purpose | Dev / Build / Prod |
| --- | --- | --- |
| `cli/src/command.dart` | read `@Flag` / `@Option` on command classes | every CLI invocation |
| `runtime/src/mirror_context.dart` | discover `Compiler` subclasses | build |
| `runtime/src/mirror_coerce.dart` | type coercion fallback | dev |
| `core/src/runtime/impl.dart` + `orm/entity_mirrors.dart` | reflect on `Controller`/`ManagedObject`/`Serializable` to drive dev mode and emit codegen | dev + build |
| `config` | reflect on `Configuration` fields | dev (with AOT compile path too) |
| `core/src/utilities/mirror_helpers.dart` | metadata helpers | dev |

The AOT build path deflects mirrors out of `core` / `config` / etc. but the
CLI itself still uses mirrors unconditionally — a real cost on every cold
invocation.

### 4.2 Build-time codegen pipeline

One pipeline, multiple plugins:

```
BuildManager
  └─ Build.execute()
       ├─ enumerate Compiler subclasses (via MirrorContext)
       ├─ for each: getUrisToResolve → CodeAnalyzer resolves AST
       ├─ for each SourceCompiler: compile() → emits Dart source
       ├─ RuntimeGenerator writes generated_runtime/ package
       ├─ for each Compiler: deflectPackage() → copy + rewrite
       └─ emit final pubspec, run AOT
```

Generator uses string-token replacement
([packages/runtime/lib/src/generator.dart](packages/runtime/lib/src/generator.dart))
rather than a `code_builder`-style AST. Isolate executables hand-serialize
results.

### 4.3 Global / implicit state

- `RuntimeContext.current` — global runtime registry.
- `ManagedDataModel` uses `fromCurrentMirrorSystem()` to populate a
  process-wide entity registry.
- `Controller.includeErrorDetailsInServerErrorResponses`,
  `Controller.letUncaughtExceptionsEscape` — static toggles.
- Many logger instances are statics.

### 4.4 Reflection-based DSLs

- **Query where clauses**: `query.where((o) => o.foo)` works by analysing
  the closure to recover the property name. Performance and refactor-
  safety hazard; any rename of a field is load-bearing on the closure.
- **@Bind and @Operation**: name-based parameter binding, path-variable
  count hardcoded to ≤4.
- **OAuth scopes**: space-delimited strings.

## 6. Refactor opportunities (ranked)

Ordered roughly by leverage / blast radius. Each item names the concrete
target; none of them are prescriptions.

### High-leverage

1. **Remove mirrors from the CLI argument layer.**
   [packages/cli/lib/src/command.dart](packages/cli/lib/src/command.dart)
   runs reflection on every command invocation to read `@Flag` / `@Option`.
   A codegen pass or a hand-written dispatch table would speed cold start
   and let the CLI ship as a plain AOT binary without the mirror runtime.
   Low risk: command inventory is small and stable.

2. **Collapse dev-mode mirror runtimes behind a smaller surface.**
   `core/src/runtime/impl.dart`, `orm/entity_mirrors.dart`, `orm_impl.dart`,
   `resource_controller_generator.dart` each do their own reflection. A
   single introspection module per artefact (Channel, Controller, Entity,
   Serializable) with one mirror entry point would make it easier to swap
   in an `analyzer`-driven pipeline later.

3. **Kill string-prefix type dispatch in `slow_coerce.dart`.**
   [packages/runtime/lib/src/slow_coerce.dart](packages/runtime/lib/src/slow_coerce.dart)
   branches on `T.toString()` prefixes. Replace with explicit
   per-type coercers generated alongside entities/serializables, or with a
   type-token map built at codegen time.

4. **Query DSL closure analysis.**
   `query.where((o) => o.field)` works by introspecting the lambda. This
   is the single most magical thing in the framework and the biggest
   footgun for downstream users. Investigate: macros (once stable), or a
   column-selector object (`Q.of<User>().name`).

5. **Extract codegen scripts from `packages/cli` into a small
   `conduit_build` package.** Scripts in
   [packages/cli/lib/src/scripts/](packages/cli/lib/src/scripts/) import the
   entire framework because they need ORM / OpenAPI / Channel types inside
   isolates. Splitting them out makes the CLI proper a thin dispatcher and
   makes it possible for third parties to trigger builds from their own
   tooling.

### Medium-leverage

6. **Decompose `Application`.** Lifecycle + isolate supervisor +
   server wrapper + options container are all tangled. Targets:
   [packages/core/lib/src/application/application.dart](packages/core/lib/src/application/application.dart),
   `isolate_supervisor.dart`, `isolate_application_server.dart`.

7. **Decompose `ResourceController`.** Operation discovery, parameter
   binding, execution, documentation live in the same class
   ([packages/core/lib/src/http/resource_controller.dart](packages/core/lib/src/http/resource_controller.dart)).
   Splitting the discovery + binding phase into a precomputed plan (which
   codegen already wants to emit) would remove per-request reflection.

8. **Add tests for `packages/common`.** Zero coverage for a load-bearing
   interface. Even interface-contract tests would catch accidental
   signature drift.

9. **Decide on OpenAPI v2 (Swagger).**
   [packages/open_api/lib/src/v2/](packages/open_api/lib/src/v2/) is
   significant code and no caller in this repo emits it. Either mark
   deprecated + plan removal, or justify its presence with a doc note.

10. **Write the v6.0 migration guide.** There is a breaking jump from 4.x
    docs to a 6.0 code release with no upgrade path documented. Pair with
    any API changes the refactor introduces.

### Lower-leverage / housekeeping

11. Replace the template-token string replacement in
    [packages/runtime/lib/src/generator.dart](packages/runtime/lib/src/generator.dart)
    with `code_builder`.

12. Hardcoded limits: 4 path variables in `@Operation`
    ([packages/core/lib/src/http/resource_controller_interfaces.dart](packages/core/lib/src/http/resource_controller_interfaces.dart)
    — verify).

13. Migration discovery is regex-on-filename (`^[0-9]+[_a-zA-Z0-9]*\.migration\.dart$`)
    in [packages/cli/lib/src/mixins/database_managing.dart](packages/cli/lib/src/mixins/database_managing.dart).
    No uniqueness / contiguity validation.

14. `ConduitCompiler.didFinishPackageGeneration` has hardcoded package
    names and paths
    ([packages/core/lib/src/runtime/compiler.dart](packages/core/lib/src/runtime/compiler.dart)).

15. Audit `packages/core/lib/src/utilities/sourcify.dart` — purpose is not
    obvious from the name or docs.

16. Global `DataModelManager` registry is mutable and populated on first
    `ManagedDataModel` construction.
