# Migrating to Conduit 6.0

Conduit `6.0.0` was published on 2026-03-09. This guide covers the
upgrade path from a 4.x or 5.x series. Everything here describes the
6.0.0 cut on pub.dev — for changes still cooking on `master` (the
`build_runner`-driven AOT pipeline that retires `conduit build`),
see the bottom of this page.

## Quick checklist

```yaml
# pubspec.yaml
environment:
  sdk: ">=3.11.0 <4.0.0"

dependencies:
  conduit: ^6.0.0
  conduit_postgresql: ^6.0.0

dev_dependencies:
  conduit_test: ^6.0.0
```

```sh
dart pub upgrade
```

The application code that compiled against 4.3.6 / 5.x typically
compiles against 6.0.0 unmodified. Read on for the cases that do
require a touch.

## 1. Dart SDK floor — `>=3.11.0`

Bump the `environment.sdk` constraint in every package's `pubspec.yaml`.
6.0.0 was published against Dart 3.11; older SDKs (3.4 from the
previous series) will fail to resolve.

## 2. Package surface

If you upgraded from a release earlier than 4.3.6 you will need to
add the separated packages explicitly:

| Replaces | Now lives in |
| --- | --- |
| `conduit` (everything bundled) | `conduit` + `conduit_postgresql` |
| `conduit_common_test` (removed in PR #185) | `conduit_test` (or `package:conduit_core/test.dart`) |

There is no longer a `conduit_common_test` package. If you imported
helpers from it, switch to `package:conduit_test/conduit_test.dart`.

## 3. `ServiceRegistry` is gone — use Dart `Finalizer` if you need it

Replaced in [PR #185](https://github.com/conduit-dart/conduit/pull/185)
("Replace ServiceRegistry with Finalizers"). The class file
`packages/core/lib/src/application/service_registry.dart` was deleted;
any imports of `ServiceRegistry` will fail to compile.

If your code registered services for cleanup at application shutdown,
move that cleanup into your channel's `close()` override:

```dart
// Before — 4.3.6 / 5.x
class MyChannel extends ApplicationChannel {
  late MyService service;

  @override
  Future prepare() async {
    service = MyService();
    ServiceRegistry.defaultInstance.register(service, (s) => s.dispose());
  }
}

// After — 6.0
class MyChannel extends ApplicationChannel {
  late MyService service;

  @override
  Future prepare() async {
    service = MyService();
  }

  @override
  Future close() async {
    await service.dispose();
    await super.close();
  }
}
```

If you need finalizer-on-GC semantics specifically (rare in server
apps), use Dart's standard `Finalizer` from `dart:core`. The framework
no longer provides a registry helper.

Two doc comments still mention `ServiceRegistry` (in
`packages/core/lib/src/application/{application,channel}.dart`); they
are stale and slated for removal. Ignore them.

## 4. Snake-case naming is opt-in (non-breaking by default)

[PR #153](https://github.com/conduit-dart/conduit/pull/153) added
opt-in snake_case naming for tables, columns, and JSON keys. Defaults
remain camelCase to preserve existing schemas:

```dart
@Table(useSnakeCaseName: true, useSnakeCaseColumnName: true)
class _User {
  @primaryKey
  int? id;

  // Persists as `display_name` instead of `displayName`.
  String? displayName;
}

class User extends ManagedObject<_User> implements _User {
  // Serializes as `created_at` in JSON instead of `createdAt`.
  @ResponseKey(name: 'created_at')
  DateTime? createdAt;
}
```

For new tables: turn the flags on. For tables that already exist in
production: leave the flags off — turning them on mid-stream causes
the schema differ to emit a rename for every column. If you do want
to migrate an existing schema, generate and review the migration
explicitly with `conduit db generate`.

## 5. Test fixture connection details

The `PostgresTestConfig` helper used by every DB test now resolves
each connection setting in this order:

1. Environment variable (`POSTGRES_HOST`, `POSTGRES_PORT`,
   `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB`).
2. Default matching `ci/docker-compose.yaml`:
   - host `localhost`, port `15432`
   - user `conduit_test_user`, password `conduit!`, db `conduit_test_db`

To run the test suite locally:

```sh
docker compose -f ci/docker-compose.yaml up -d
melos run test-unit
```

Earlier `6.0` development builds threw `ArgumentError` if any of the
five env vars were missing. If you wrote a `.settings.yaml` shim to
work around that, you can delete it.

## 6. Smaller things you may notice

- `packages/cli/lib/src/utilities/{file_system,lowercasing_map,mirror_helpers,reference_counting_list,sourcify,text_case,token_generator}.dart`
  were deleted in PR #185 (internal utilities; not part of the
  public API).
- `package:conduit/conduit.dart` re-exports unchanged, so most user
  code is unaffected by the file-level reshuffles.
- `analyzer ^10.0.0` is the new minimum (was `^5`).

## What's coming on `master` (not in 6.0.0)

The `master` branch carries the AOT-via-`build_runner` migration
described in `docs/AOT_WITHOUT_BUILD.md`. The big changes in flight:

- `conduit build` is retired; the user-facing AOT path is
  `dart run build_runner build && dart compile exe bin/main.dart -o build/server`.
- A new `conduit_build_runner` package emits per-class runtimes at
  build time, replacing the runtime mirror walk. The mirror code path
  remains as a JIT fallback through one release cycle.
- `conduit_runtime` splits into `runtime.dart` (lean, mirror-free)
  and `dev.dart` (mirror-using, dev-only).
- `conduit_core` adds an `aot.dart` entrypoint that excludes the
  mirror-using `Compiler` re-exports.
- SDK floor moves to `>=3.12.0-0`.

These ship in a 6.x follow-up, not in 6.0.0. If you're tracking
`master`, see `docs/AOT_WITHOUT_BUILD.md` and `docs/REFACTOR_P6_NOTES.md`
(the latter is on the `refactor-p6-notes` branch on forgejo).
